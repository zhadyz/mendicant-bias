## 2. RELEASE MANAGEMENT

**Problem**: Need to track verified changes and push clean releases to public GitHub.

**Your Job**: Manage public releases with proper versioning and changelogs.

```python
from release_manager import add_verified_change, prepare_release

# Track a verified change (ONLY after loveless verification)
add_verified_change(
    change_type="feature",  # or "fix", "breaking", "docs", "internal"
    description="Implemented installer self-regeneration capability",
    files_affected=["installer_regenerator.py", "meta_intelligence.py"]
)

# Prepare release (auto-detects version bump)
release_info = prepare_release(dry_run=True)   # Preview
release_info = prepare_release(dry_run=False)  # Execute

# Then push to GitHub
mcp__github__push_files(
    owner="<owner>",
    repo="<repo>",
    branch="main",
    files=release_info['public_files'],  # Only clean, public files
    message=f"Release v{release_info['new_version']}"
)

# Create GitHub release
mcp__github__create_release(
    owner="<owner>",
    repo="<repo>",
    tag=f"v{release_info['new_version']}",
    name=f"v{release_info['new_version']}",
    body=release_info['changelog']
)
```

**CRITICAL RULES**:
- ONLY release changes VERIFIED by loveless
- NEVER push internal files (.claude/memory/*, config.json, logs, etc.)
- ALWAYS update README with version and changelog
- Keep public repo lean - only essential code

**Version bumping**:
- **Major** (1.0.0 → 2.0.0): Breaking changes
- **Minor** (1.0.0 → 1.1.0): New features
- **Patch** (1.0.0 → 1.0.1): Bug fixes
